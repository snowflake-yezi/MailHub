### 接口流向

| # | 方向 | 说明 |
|---|------|------|
| ① | 出票中心 → 管理系统 | 创建邮箱：`POST /api/v1/mailboxes` |
| ② | 管理系统 → 邮箱服务器 | 创建邮箱账户/别名；查询邮件 |
| ③ | 大模型系统 → 管理系统 | 获取邮件原件：`GET /api/v1/orders/{orderId}/emails` |
| ④ | 邮箱服务器 → union 邮箱 | 自动转发关键邮件 |

## 2. 管理系统（Management System）

### 2.1 职责

```
管理系统 = 控制面（Control Plane）
```

| 职责 | 说明 |
|------|------|
| **邮箱生成 API** | 暴露给出票中心，一订单一邮箱，返回分配好的邮箱地址 |
| **服务器池管理** | 注册/注销邮箱服务器，维护服务器状态（健康/故障/满载） |
| **分配策略** | 决定新邮箱落在哪台服务器（轮询/最少连接/指定规则） |
| **映射存储** | order_id ↔ email_address ↔ server_id 的映射关系持久化 |
| **邮件查询 API** | 暴露给大模型系统，按订单 ID 查询邮件原件（透传到底层服务器） |
| **配置管理** | 管理过滤规则、转发目标、白名单/黑名单 |
| **监控告警** | 服务器健康检查、邮件量统计、API 调用量 |

### 2.2 核心数据

```
order_mailboxes 表：
  order_id        → 订单号
  email_address   → 生成的邮箱地址
  server_id       → 所在服务器
  created_at      → 创建时间
  expired_at      → 过期时间（订单完成后回收）
  status          → active / expired / disabled

mail_servers 表：
  server_id       → 服务器唯一标识
  host            → 内网地址
  port            → API 端口
  status          → healthy / degraded / down
  capacity        → 最大邮箱数
  current_load    → 当前邮箱数
  last_heartbeat  → 最后心跳时间
```

### 2.3 对外 API

```
POST   /api/v1/mailboxes              → 出票中心调用，创建邮箱
GET    /api/v1/mailboxes/{order_id}   → 查询某订单的邮箱地址
DELETE /api/v1/mailboxes/{order_id}   → 订单完成后回收邮箱

GET    /api/v1/orders/{order_id}/emails   → 大模型系统调用，获取邮件原件列表
GET    /api/v1/emails/{email_id}/body     → 获取单封邮件完整正文（含附件元数据）

POST   /api/v1/servers                → 管理后台：注册新邮箱服务器
GET    /api/v1/servers                → 查看服务器池状态
```

## 3. 邮箱服务器（Email Server Node）

### 3.1 职责

```
邮箱服务器 = 数据面（Data Plane）
```

| 职责 | 说明 |
|------|------|
| **SMTP 收信** | 接收外部发往 `@ticket.xxx.com` 的邮件 |
| **邮件存储** | 以 IMAP/Maildir/DB 形式存储邮件 |
| **过滤引擎** | 按规则过滤广告/垃圾邮件 |
| **自动转发** | 关键邮件转发到 union@xxx.com，标题带源地址前缀 |
| **邮件查询** | 暴露内部 API 给管理系统，按邮箱地址查询邮件 |
| **心跳上报** | 向管理系统上报健康状态和负载 |

### 3.2 对内 API（管理系统 ↔ 邮箱服务器）

```
POST   /internal/mailboxes            → 创建邮箱（管理系统的②调用）
DELETE /internal/mailboxes/{address}  → 删除邮箱
GET    /internal/mailboxes/{address}/messages  → 获取邮件列表
GET    /internal/messages/{id}        → 获取单封邮件完整内容
GET    /internal/health               → 健康检查
GET    /internal/stats                → 负载统计
```

### 3.3 邮件处理流水线

```
SMTP 收信 → [垃圾过滤] → [关键邮件判定] → [存储] → [转发到union]
                                        ↓
                              管理系统按需查询 → 大模型系统
```

## 4. 大模型系统（外部，不在本项目范围）

我们的系统只负责提供数据，不调用 LLM：

```
大模型系统 → GET /api/v1/orders/{orderId}/emails → 管理系统 → 透传到邮箱服务器
          ← 返回邮件原件（发件人、时间、标题、正文、附件）
          
大模型系统自行处理：
  - 邮件分类（出票/退改签/航变）
  - 信息抽取
  - 模版填充
```

## 5. 关键设计约束

| 约束 | 说明 |
|------|------|
| **管理系统 1:N 邮箱服务器** | 管理系统是单点（可做主备），邮箱服务器可横向扩展 |
| **邮箱服务器无状态倾向** | 邮箱地址的分配由管理系统决策，服务器只负责执行，便于替换和扩容 |
| **大模型系统解耦** | 邮件原件通过 API 拉取，不在本项目做 LLM 调用 |
| **union 邮箱是最终汇聚点** | 所有关键邮件转发到这里，运营人员在此统一查看 |
